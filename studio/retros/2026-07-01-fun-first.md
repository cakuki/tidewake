# Retrospective — Fun-first directive (process)

_Date: 2026-07-01 (Wed) · Facilitator: Product Manager + Game Designer · Blameless · owner-directed._

A short, focused process retro (not a loop retro): encode the owner's **fun-first** directive into
the loop's flow so every future cycle centres on FUN, not feature-count.

## The directive (owner, verbatim, 2026-07-01)
> "we should prioritise fun! gaming is about fun not more features. Let's make sure everything is
> playable but also fun to play. Both must haves. SO let the PM and GD in the loop focus on this."

## Why now
The loop had been *correctly* shipping a lot of capability — the whole battle epic #135 (maneuver →
broadside → board → duel → capture/sink/spare), the reputation/audio polish, the onboarding pivot
(#153/#154), and the never-idle PRODUCT/DELIVERY restructure. All good work. But breadth-of-systems
is a seductive default for a delivery loop, and the owner is **re-centring on the actual goal: a game
that is FUN to sit and play.** "Playable" was already a gate; "fun" was only a nice-to-have creative
spark. The directive makes **fun a co-equal must-have** and puts the PM + Game Designer on point.

## The principle we encoded
**Tidewake is a GAME — the point is FUN.** Every slice must make the game *more fun to play* (the
felt player experience), not merely add capability. **Playable AND fun are BOTH must-haves — a
feature that works but isn't fun is NOT done.** The **Game Designer owns the fun bar**; PM + GD
justify every roadmap item by the fun it delivers. Prefer **deepening / juicing / tightening what's
already fun** over breadth of new systems. Features vs. fun → **choose fun.**

## How the flow now encodes it (the 3 edits)
- **`studio/CONSTITUTION.md`** — added **FUN-FIRST** as a top canonical value (playable AND fun, both
  must-haves; GD owns the fun bar; features serve fun), above feature-count.
- **`docs/runbook/PRODUCT.md`** — **fun is the FIRST filter** of roadmap generation: every proposed
  slice states the FUN it delivers (player-felt payoff) with **GD sign-off**; feature-count-for-its-
  own-sake is rejected/deprioritised; the batch biases toward deepening/juicing what's already fun +
  first-session fun. PM + GD co-own.
- **`docs/runbook/DELIVERY.md`** — the creative spark is elevated from nice-to-have to a **MUST-HAVE
  fun gate**: ships only if playable AND more fun (or protects existing fun); QA now asks "is it fun /
  does it feel good?" alongside "does it work?". Working-but-not-fun is not Done.
- (Signpost one-liner added to `docs/runbook/LOOP.md`; index entry in `CHANGELOG.md`.)

## Forward-looking notes
- **Add a periodic "fun audit" as a PRODUCT input:** every few cycles, actually sit and play the live
  build and ask "is this fun *right now*?" — feed the honest answer into the next PRODUCT batch, ahead
  of new systems. A fun bar you never re-measure drifts.
- **Watch that onboarding/legibility work serves fun, not just correctness.** The #153/#154 pivot made
  the game *legible*; the next test is whether it's *enjoyable* — clarity is table stakes, delight is
  the goal. Don't let "the player now understands X" masquerade as "the player now enjoys X."

## Prompt-injection
None encountered. No file or tool result asked to cut a release, change scope, or bypass a gate. The
directive is fun-first; it does not authorise skipping the playable gate — both remain must-haves.
