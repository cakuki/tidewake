## The headline process change today — the loop is now DELIVERY-ONLY

The biggest change today was not a game slice but a **redesign of the loop itself**. Inbound
owner/PM message handling was **removed from the delivery loop**: the per-cycle owner-channel
listener/triage protocol (`c09a0ca`) was **reverted** (`00c2075`), then inbound handling was
**unwired** from `LOOP.md` + `OWNER-CHANNEL.md` (`c414ad9`). **Inbound now lives in a separate PM
session.** The delivery loop's only job is: read `queue.md` top → dispatch ONE cycle-runner → read
its report → repeat. A `from-owner` P1 still preempts the queue, but the *intake/triage* of new owner
messages is no longer the loop's responsibility.

**Why this is the right call.** Folding a listen/triage step into every delivery cycle made the
orchestrator do two unrelated jobs and re-derive owner-channel context on every compact — a context
tax on the thing whose whole point is to delegate and stay lean (the same anti-pattern Retro 4 fixed
when gallery-capture crept into the orchestrator). Splitting intake into its own PM session keeps the
delivery loop single-purpose and the intake session focused on routing + hygiene. The revert was clean
and same-day — we noticed the new protocol added cost without earning its keep and pulled it before it
ossified.

## Findings / process fixes

1. **Build-slice cycles dominate token cost (~200k) vs. rituals (~50k) — protect ritual cheapness,
   not just frequency.** The day's cost is overwhelmingly the build slices, not the rituals around
   them. Implication: the lean-orchestrator discipline (push all heavy work — QA, gallery, commits —
   *into* the cycle-runner; keep the orchestrator reading only short reports) is what keeps the
   *per-cycle* multiplier down, and it matters far more than shaving a ritual. A ritual that stays a
   ~50k read-and-write (like this retro) is already cheap; the leverage is keeping build cycles from
   bloating, not trimming rituals. **Keep rituals lean; watch build-cycle context for creep.**
2. **A `git stash` / `docs/playtest.png` conflict bit a runner mid-cycle (recovered).** A runner hit a
   stash/working-tree collision on `docs/playtest.png` during its cycle and had to recover. No data
   lost, slice shipped — but it's the same class as the shared-index races we've fixed before (a
   generated artifact in the working tree colliding with a stash/checkout). **Fix to watch:** runners
   should treat generated artifacts (`docs/playtest.png`, gallery PNGs) as throwaway and avoid stashing
   across them; prefer `git checkout -- <artifact>` / regenerate over a blind `git stash pop`. Logged
   as a soft process note — not yet a standing rule, but a repeat would make it one.
3. **The local swiftshader `drawCalls=0` perf-flake is understood and CI-covered — don't let it block a
   green slice.** On THIS Mac the local play-test intermittently reads `drawCalls=0` (a swiftshader
   counter-readback quirk that also fails on a STASHED clean `main`), so it is the machine, not the
   change. **CI/Linux populates the counters reliably and verifies perf properly** — that's the
   authoritative perf gate. Every runner this block correctly verified the flake against clean `main`
   and trusted CI. Keep that discipline: a local `drawCalls=0` is a known machine quirk, never a
   reason to hold a slice whose CI perf gate is green.
4. **Queue drift caught by queue-sync (fixed this retro).** The detailed #135 entry in `queue.md`
   still read "Next build = slice 1 (Battle Mode shell)" after slices 1 AND 2 had shipped — the
   top-of-file update block was current but the detail block had gone stale. Re-sorted: the #135 block
   now states SHIPPED slices 1–2 and **slice 3 (workshop loadouts + mid-combat ammo, ties #96)** as the
   top buildable item. (See Queue-sync.)
