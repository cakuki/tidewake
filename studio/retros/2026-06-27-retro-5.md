## Studio / process review

**What's reliable now (keep).**
- **Cycle-runner subagents that self-ship are the workhorse** — 7 clean slices this block, gate
  green throughout, zero escaped prod bugs, owner P1s drained same-block.
- **The from-owner P1 fast-lane (Retro 4) worked first time** — the owner filed P1s through the PM
  Desk and they shipped ahead of features, exactly as designed.
- **The Game-Designer-owned tuning pass (Retro 4) delivered #57** — the curve got *tuned against a
  real session*, not guessed. The balance-owner change paid off immediately.
- **Measurement-first perf (#52)** — ceilings derived from a real measurement, asserted on
  deterministic counters (not fps), with headroom. A model for "gate the thing, don't vibe it."

**What still hurts / the headline finding.**
1. **The orchestrator still does too much per-cycle bookkeeping and live QA — and a compact makes
   that expensive.** The owner's explicit ask this session: **keep the main context lean so loops
   after a compact are cheap.** Retro 4 moved *visual QA* into the cycle-runner; this retro finishes
   the job: **cycle-runners own ALL bookkeeping** (self-commit specific files + push + verify CI,
   self-close the issue, self-append their own loop-log row, self-QA, report <10 lines). The
   orchestrator's per-cycle job shrinks to **read the top of a queue → dispatch one runner → read its
   short report**. Codified as the new **"Lean orchestrator protocol (post-compact)"** in `LOOP.md`,
   backed by a new `studio/comms/queue.md` priority queue.
2. **Process lessons learned the hard way this session — bake them in:**
   - **Cycle-runners must `git add <specific files>`, never `git add -A`** — a concurrent docs
     subagent's files got swept into a cycle-runner's commit. **Never run a docs subagent concurrent
     with a `git add -A` cycle-runner.**
   - **A few subagents transiently glitch with 0 tool-uses → re-dispatch** (already a Retro 3 rule;
     it recurred — keep enforcing).
   - **Live-browser QA needs a cache-busting reload (`ignoreCache`)** — ES modules cache, so a stale
     bundle fooled a QA pass.
   - **PARK the live WebGL tab on `about:blank` after QA** — a running render loop heats the owner's
     machine. Lean on headless/puppeteer; touch live Chrome only for owner-facing *visual* changes.
   - **QA must wait real wall-clock for CSS fades** — synchronous `tw.step()` doesn't advance them
     (re-confirmed; the Retro 3 rule holds).
3. **Owner-decisions must never be auto-adopted** — #56 (mobile) and #58 (weather, vs. the sunny
   vibe) are decisions to *ask*, not slices to *do*. The queue must mark them as such.
4. **Deep-learning loop #2 is 8 cycles overdue.** The ritual slipped under owner-sprint pressure.
   The lean protocol must schedule rituals as subagents so they stop slipping.

**Creativity (process self-review).** This was a *responsive* block — we served the owner's eye well
(sunny water, clean compass, reachable curve). The creative risk now is the opposite: serving
feedback so closely we stop *surprising*. The deep-learning refill (#2) and a depth beat with real
drama (cannon #59) are the guard against a purely reactive studio.

- **The Game-Designer tuning pass** — turned the un-tuned arc *reachable* (#57, `LEGEND_AT 2400`).
- **Measurement-first perf gate (#52)** — ceilings from a real measurement, asserted deterministically.

## What didn't (improve)
- **Per-cycle bookkeeping + live QA still partly in the orchestrator** — expensive across a compact;
  fixed by the lean protocol (cycle-runners own all of it).
- **`git add -A` in a cycle-runner swept up a concurrent docs subagent's files** — never again;
  specific-path adds + no concurrent docs-during-add-A.
- **Live WebGL tab left rendering heats the machine** — park on `about:blank` after QA.
- **Deep-learning loop #2 slipped ~8 cycles** — rituals must be scheduled as subagents, not deferred.
