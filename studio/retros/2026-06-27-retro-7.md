## Studio / process review

  fix was verified when it can't be checked headlessly.

**What hurt / the findings.**
1. **The #86 "ocean void" misdiagnosis — a cycle chased a phantom.** A single owner screenshot (a
   zoom-in close-up of the hull) was read as "the ocean isn't rendering on iOS," and Loop 34 shipped
   shader-precision/fallback hardening for a **non-bug** — the owner later clarified the sea was fine.
   The hardening was harmless (and arguably good defensive code), but a whole cycle was spent on a
   defect that didn't exist, and the owner's real signal was misunderstood. **Root cause:** an
   owner-reported *visual* bug was acted on **without the orchestrator fetching and actually VIEWING
   the full-frame screenshot** and reconciling it against expected behaviour. **Fix:** for any
   owner-reported visual bug, the orchestrator must **fetch the real screenshot, view the FULL frame,
   confirm the bug is real** (and ask for a wider/clarifying shot if the capture is ambiguous) **before
   dispatching a fix.** A close-up crop is not a diagnosis.
2. **Owner-sent media wasn't trivially accessible.** To even retrieve the owner's photo, the
   orchestrator had to hand-roll Telegram Bot API calls (`getFile` + a token-bearing file download from
   `config.env`). Seeing what the owner sent should be a **first-class, one-command capability**, not
   an ad-hoc API spelunk every time. **Fix (the headline improvement this retro): a new
   `scripts/owner-channel.sh photo [--latest|--id <file_id>] [--out <path>]` subcommand** — reads the
   token from config (never echoed; curl `-K -` keeps it out of argv), finds the newest UNREAD owner
   photo (peek semantics — non-consuming, so a later `inbox` still delivers it) or a specific file_id,
   `getFile` + downloads it to a local path, and **prints the path** so any cycle can open and SEE it.
   Owner-locked in spirit (only photos from the configured owner+chat). Clean "no photo" exit 3.
3. **"iOS/device fixes can't be verified locally" was practiced but not written as a rule.** The loop
   did the right thing (best-effort ship, marked pending owner re-test, didn't pile dependent work on
   it), but nothing in the runbook *says* to. **Fix:** add a guardrail — device/iOS-specific fixes ship
   best-effort, are tracked as **unconfirmed pending owner re-test**, and the loop must **not stack
   dependent work** on an unconfirmed device fix (so a phantom doesn't cascade).
4. **(Watch, not act) The release workflow's path filter doesn't match our stated policy.** The
   runbook says "releases trigger only on `src/`/`index.html`," but `.github/workflows/release.yml`
   uses **`paths-ignore`** (`studio/**`, `docs/**`, `**/*.md`) — so a push touching `scripts/**`,
   `tests/**`, `package.json`, `manifest.webmanifest`, or `.github/**` **does** trigger a release.
   That's why this very retro's `scripts/owner-channel.sh` change would have burned a release run
   (avoided here with a `[skip ci]` commit). Not fixed unilaterally (CI-trigger semantics is an
   owner-ish call); flagged for a follow-up decision (add `scripts/**` to `paths-ignore`, or switch to
   an allow-list `paths:`).

**Creativity (process self-review).** Two reactive blocks in a row (collision + platform + bug
triage). That's the right service while the owner field-tests, but the studio's *authored* surprise is
banked in the DL #2 backlog. Once the iOS reports settle, the highest-leverage creative act is pulling
a charm slice (#78 Ballad / #79 False Colours) rather than another noun or another polish pass.
